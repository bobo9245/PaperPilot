### 1. 왜 중요한가

- 두 번째 새로움은 코드 재사용을 ‘기능적으로 비슷한 코드 검색’이 아니라 ‘구현 논리까지 고려한 재사용’으로 확장한 점이다. 제공된 근거는 기존 검색 증강 방식이 유사 코드 조각을 찾아도 실제 저장소에 정확히 대응하는 구현이 없으면 성능을 해칠 수 있다고 설명한다.
- TICoder는 잠재 호출 함수 검색에서 기능 관점과 구현 관점을 함께 보는 이중 관점 유사도를 사용한다. 이는 함수가 무엇을 하는지뿐 아니라 저장소 안에서 어떻게 구현되고 호출되어야 하는지까지 고려하려는 차별점이다.
- 사용 패턴 선택에서도 차별점이 있다. 단순히 검색된 함수를 문맥으로 넣는 것이 아니라, 구조 기반 군집화와 퍼플렉서티 기반 필터링을 결합해 관련 사용 패턴을 고르는 이중 단계 전략을 사용한다고 제시된다.
- 기존 접근 대비 포괄성도 새로움으로 제시된다. 제공된 비교 설명에 따르면 TICoder는 계획, 테스트 케이스, 검색, 사용 패턴을 모두 활용하는 반면, 열거된 기존 방법들은 이 요소들을 모두 동시에 갖추지는 않은 것으로 제시된다.
- 논문은 실제 개발에서는 하나의 매우 유사한 코드 조각을 그대로 재사용하기보다 여러 부분적으로 관련된 함수를 조합해 새 기능을 구현하는 경우가 많다고 전제한다. TICoder의 설계는 이 관찰을 반영해 저장소 내 호출 함수와 사용 패턴을 더 적극적으로 활용하려 한다.

### 3. 방법

방법:
- TICoder의 전체 설계는 저장소 수준 코드 생성에서 계획 수립과 코드 재사용을 결합하는 구조다. 복잡한 의존성과 제한된 문맥 창 때문에 저장소 전체를 그대로 모델에 넣기 어려운 상황을 전제로 한다.
- 테스트 주도 반복 계획 수립은 자연어 요구사항만으로 계획을 만들지 않고, 테스트 케이스를 기대 동작의 명세로 사용한다. 이를 통해 생성된 구현 단계가 실제 기대 동작과 더 잘 맞도록 반복적으로 정제한다.
- 구현 인식 호출 함수 검색은 잠재적으로 재사용할 수 있는 저장소 내 함수를 찾는 단계다. 제공된 근거에 따르면 기능적 유사성과 구현 측면을 함께 고려하는 이중 관점 유사도를 사용한다.
- 사용 패턴 선택은 이중 단계 전략으로 설명된다. 구조 기반 군집화로 사용 패턴 후보를 조직화하고, 퍼플렉서티 기반 필터링으로 관련성 높은 사용 패턴을 선택하는 방식이다.
- 이 접근은 단일 유사 코드 조각을 그대로 재사용하는 관점보다, 실제 개발자가 여러 부분적으로 관련된 함수를 조합해 새 기능을 구현한다는 관찰에 가깝게 설계되었다.
- 저장소 함수 재사용에서는 함수 호출 자체뿐 아니라 매개변수 설정 같은 구현 세부가 중요하다고 본다. 예시로 민감 키를 재귀적으로 가리는 작업에서 redact_mode 같은 매개변수 구성이 중요하다는 설명이 제시된다.

- 내적 타당성 위협이 별도 항목으로 언급되지만, 구체적인 위협 내용은 제공된 근거만으로 충분히 확인되지 않는다.

### 2. 핵심 기여

새로움/차별점:
- 가장 중요한 새로움은 RAG의 역할을 ‘외부 지식 접근’에서 ‘재사용 가능한 능력 획득’으로 확장하려는 점이다. 제공된 근거는 기존 RAG가 주로 단편적인 선언적 근거를 검색해 에이전트가 절차를 반복적으로 재구성해야 한다고 설명하며, Anything2Skill은 외부 지식에서 스킬을 추출해 에이전트에 설치하는 문제를 제기한다.
- 기존 방식과의 차별점은 외부 지식을 추론 시점의 문맥으로만 쓰지 않고, 문서·매뉴얼·대화·로그·궤적을 구조화된 스킬 계약으로 변환한다는 점이다. 이는 같은 절차를 매번 문단에서 추론하는 대신, 명시적 절차 지식으로 축적해 재사용하게 만든다는 점에서 중요하다.
- 스킬 트리 사전 구조를 사용하는 계획-확장식 추출 절차도 차별점으로 제시된다. 제공된 근거에 따르면 각 기록을 근거 창으로 분해한 뒤, 스킬 트리 기준 아래에서 절차 지식을 식별하고 스킬 초안으로 변환한다. 이는 단순 문서 검색이나 비구조적 메모리 저장과 구분된다.
- 스킬 계약의 설계가 새롭다. 스킬을 단순 설명문이 아니라 호출 조건, 금기 조건, 행동 단위, 작업 흐름, 제약, 출력 명세, 지원 근거, 신뢰도까지 포함하는 계약 형태로 표현해, 에이전트가 언제 어떻게 스킬을 써야 하는지 더 명확하게 안내하도록 한다.
- SkillBank라는 지속적 절차 기억 관리 구조도 주요 차별점이다. 제공된 근거는 스킬을 단순 인덱스에 넣는 것이 아니라 분류체계 기반 컴파일, 등록 수준 조정, 생명주기 추적, 버전 갱신, 스킬 트리 투영으로 관리한다고 설명한다.
- 기존 스킬 기반 에이전트와 비교해, 수작업으로 정한 스킬이나 환경 상호작용으로 학습한 스킬, 과제 수행 중 생성한 실행 코드에만 의존하지 않고, 이미 존재하는 이질적 지식 기록에서 재사용 가능한 절차 스킬을 컴파일한다는 점을 차별점으로 내세운다.
- 평가 관점의 새로움은 컴파일된 스킬이 단순한 추가 검색 색인이 아니라, 과제 시점 문서 검색 없이도 에이전트를 안내할 수 있는 절차 지식인지 확인하려 한 점이다. 제공된 근거는 Anything2Skill 단독이 기본 에이전트보다 높은 성공률을 보였고, RAG와 결합했을 때 가장 좋은 결과를 냈다고 설명한다.

주요 기여:
- 이 논문은 검색증강생성(RAG)이 추론 시점에 외부 지식을 가져오지만, 주로 단편적인 선언적 근거를 제공해 에이전트가 매번 절차를 다시 추론해야 한다는 문제를 제기한다.
- 핵심 기여는 이질적인 외부 지식 기록을 재사용 가능하고 검색 가능하며 실행 가능한 ‘스킬’로 컴파일하는 Anything2Skill 프레임워크를 제안한 것이다.
- 문서, 매뉴얼, 대화, 실행 로그, 궤적 같은 다양한 지식원을 단순 검색 대상이 아니라 절차적 능력의 원천으로 다룬다는 점을 주요 기여로 제시한다.
- 추출된 스킬을 호출 조건, 금기 조건, 행동 단위, 작업 흐름, 제약, 출력 명세, 근거, 신뢰도 등을 포함하는 구조화된 스킬 계약으로 변환한다.
- 스킬을 지속적으로 관리하기 위해 SkillBank를 구성하고, 분류체계 기반 컴파일, 등록 수준 조정, 생명주기 추적, 버전 갱신, 스킬 트리 투영을 수행한다고 설명한다.
- 추론 시점에는 기존 지식 베이스에서 과제별 문단을 검색하는 RAG와 SkillBank에서 절차적 스킬을 검색하는 방식을 결합해, 선언적 근거와 절차적 안내를 함께 제공한다.
